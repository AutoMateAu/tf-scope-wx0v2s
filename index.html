02 · Filter rules

Four rules, applied in order.

  • Four deterministic filters on raw Asana data.
  • Remaining candidates handed to the LLM for the final 3–5 selection.

Rule 1 · Read ownership from subtask names

  • Pattern: ↳ (name) to review|approve|sign|check|call|send|look|confirm.
  • When it fires, subtask is owned by the named person — regardless of assignee field.
  • Catches 11 Ryan items (vs 7 via assignee-only).
  • Verb list is a Notion config DB Ryan edits — not code.

Rule 2 · Drop SOP checklist subtasks

  • Many recurring parents (payroll, BAS, bank rec) have step-by-step subtasks: ↳ Login to Xero, ↳ Click Payrun, ↳ Send Payslips.
  • Instructions, not work. Heuristic: no due date + no assignee + ≥5 sibling subtasks.
  • Drops 914 of 1,022 open subtasks.
  • Rule 1 runs first — ownership-tagged subtasks are exempt.

Rule 3 · Drop tasks >90 days overdue

  • Treated as uncleaned workspace cruft. Dropped silently — not surfaced anywhere.
  • Catches 6 items today, including a Jan 9 "Ryan to review" on Podia payroll (104 days).
  • Ryan can clean them up in Asana directly if he wants.

Rule 4 · Roll up person × project clusters

  • One assignee · ≥10 overdue · concentrated on 1–2 clients → one line.
  • Today: Vaishnavi · 29 overdue on EMPAC → single line with click-through.
  • 29 rows would drown the view; "EMPAC backlog, 29 items on Vaishnavi" is the actual signal.

Then the LLM ranks

  • ~30 Asana candidates + 14 email triage + 1 commitment = ~45 pooled.
  • LLM ranks across channels — email items can outrank Asana items.
  • Picks 3–5 lines.
  • Same pattern as the email triage playbook already running daily.

06 · How it runs

Integration model and a walkthrough.

  • What the playbook depends on.
  • How it talks to Notion and Asana.
  • One real 06:00 run using 23 April data.

A · Integration model

  • Runtime: Cowork (Anthropic's agentic runtime). Hosts the schedule, runs Claude calls, talks to externals through MCP connectors.
  • No backend service. No Postgres. No Lambda. No cron server.
  • If Cowork is up, the playbook runs.
System MCP connector Auth What it does on each run
Asana asana-mcp OAuth on Floyd's account · 1-hour access token, refresh token persisted in Cowork secrets Reads tasks + subtasks across Ryan's 10 projects (~118 API calls per run, well under Asana's 1,500/min limit)
Notion notion-mcp Notion integration token (the same one Iter 1 already uses) Reads config DBs (verb list, assignee map, Client Register) · reads data DBs (Email Triage — all of today's items, not just red, Commitments, Interaction Log) · writes the output blocks · writes Run Log summary · writes Decision Log rows (one per candidate, ~45/run)
  • Same MCP layer already runs email triage (live since Iter 1). Today's Focus is a second playbook on the same runtime.
  • Triage runs 07:50 → writes Email Triage DB. Today's Focus runs at 08:00 and 16:00 → reads that DB → picks items that need today.
  • Today: triage surfaced 14. Today's Focus picked 2 (Kering escalation, Podia BAS). Other 12 stay in the triage block.

B · Notion write pattern

  • Writes to one Notion page only: Ryan's Workspace (page ID 3458c003888081c5988cdf589d9d7ce2, already used for the email triage block).
  • Never creates pages. Never modifies databases. Touches one specific block on one page.
  • Marker-block pattern: heading block "🎯 Today's Focus" on Ryan's workspace page. Found by ID each run (cached after first lookup).
  • Write: one call — PATCH /v1/blocks/{marker_id}/children — atomically replaces all child blocks.
  • Crash-safe: commits fully or not at all. No half-rendered state.
  • Failure mode: yesterday's render stays until the next successful run.
  • Scope: other blocks on the page (Email Triage, Ryan's notes) untouched.

D · Decision log

  • Two Notion DBs written per run.
  • Run Log DB — one row per run. Totals: raw pulled, rules dropped, candidates, surfaced, tokens, runtime, status.
  • Decision Log DB — one row per candidate considered. Fields: run_id, candidate_id, channel, source_title, source_link, matched_rule, priority, surfaced (Y/N), llm_reasoning, run_timestamp.
  • Answers "why didn't X show up?" — query the Decision Log, find the candidate, see which rule dropped it or how the LLM ranked it.
  • Ryan-visible in Notion. Filterable by client, by run date, by surfaced / not-surfaced.
  • Feeds the tuning loop — change a verb or threshold, re-run against the same window, see what flips.
  • Retention: 90 days rolling · old rows auto-archive.
